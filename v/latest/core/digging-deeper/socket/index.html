Sockets

Warlock ships a Socket.IO connector. When src/config/socket.ts exists, the framework starts a Socket.IO server next to your HTTP server, parks the instance in the DI container, and hands you a one-liner to grab it from any controller, service, or use-case. From there youŌĆÖre using vanilla socket.io ŌĆö rooms, namespaces, middleware, all of it.

ThatŌĆÖs the honest scope today. The framework ships only the server side. The framework owns the lifecycle (boot, watch the config for changes, shut down cleanly). What you build on top ŌĆö auth handshakes, per-user rooms, broadcast helpers ŌĆö is plain Socket.IO code in your modules. The client side is plain socket.io-client (see Connecting a client). WeŌĆÖll show the patterns that fit cleanly; deeper integrations (per-socket auth resolution, a typed event bus, automatic room joining for logged-in users) are roadmapped, not shipped.

socket.io is an optional peer. The framework does not depend on it directly ŌĆö the connector lazy-imports it on boot. If you add src/config/socket.ts without socket.io installed, the connector throws on startup with install instructions. Run warlock add socket to install the dependency and eject a starter config.

The HTTP side of Warlock covers request/response. Sockets cover the cases HTTP canŌĆÖt:

  • Push to the client. Order status changes, new chat messages, agent progress updates.
  • Subscribe to live data. Dashboard widgets, presence indicators, collaborative editing.
  • Low-latency two-way. Anything where a poll loop would be silly.

If your need is ŌĆ£the user clicks a button and gets a reply,ŌĆØ HTTP is fine. If your need is ŌĆ£the server has something new to say without being asked,ŌĆØ thatŌĆÖs socket territory.

The port story matters. If your project has src/config/http.ts (which most do), the socket connector reuses the Fastify HTTP server ŌĆö no separate port, no second listener. SSL follows the same path: with HTTP enabled, you inherit its SSL setup; without HTTP, the connector creates an HTTPS server instead of an HTTP one when an ssl flag is truthy on the config.

ssl is read but not yet on the type. The connector reads socketConfig.ssl at runtime to decide HTTP vs HTTPS, but the exported SocketOptions type only declares port? and options?. If you need the standalone HTTPS path, drop the as SocketOptions annotation (or cast through as any) until the field lands on the type. With HTTP enabled this never matters ŌĆö SSL is inherited from the Fastify server.

So saving the config disconnects open sockets and shuts the old server down, but a fresh server with your new options is not stood back up until a full dev-server restart. This is a current limitation ŌĆö restart the dev server to apply config changes.

Gotchas

Section titled ŌĆ£GotchasŌĆØ
  • No socket without config. getSocketServer() returns null if src/config/socket.ts doesnŌĆÖt exist. Always null-check before emitting; donŌĆÖt litter ! non-null assertions across services.
  • Editing the socket config needs a full restart. Saving src/config/socket.ts runs the connectorŌĆÖs restart() (= shutdown() + start()), which tears the running server down.   • Listener registration leaks. Adding io.on("connection", ŌĆ”) at the top of a hot-reloaded file means each save piles on another listener. Always register in main.ts (which is auto-loaded once per boot, not per module re-import) ŌĆö or guard with onCleanup to remove the listener on reload.
